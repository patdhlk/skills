``issue`` directive, status-driven through the triage state machine
(:need:`ADR_0005`). IDs are allocated dense, max+1 (:need:`ADR_0008`).

.. issue:: pds lint — mechanized body lint
   :id: ISSUE_0013
   :status: needs-triage
   :kind: improvement

   *This was generated by AI during triage.*

   The strict gate (:need:`ADR_0007`, :need:`ADR_0017`) catches
   structural problems — schema violations, broken links, duplicate
   IDs — but says nothing about the substance of a need body.
   Implement body lint as a ``pds`` verb (:need:`ADR_0014`), runnable
   alone (``pds lint``) and as part of ``pds check``:

   - required sections per type (an ``arch-decision`` must carry
     Context / Decision / Consequences; an ``issue`` body must be
     non-trivial),
   - weasel-word check on ``req`` bodies (vague adjectives without a
     numeric or behavioral criterion),
   - unenumerated-quantifier check on ``req`` bodies (``all`` /
     ``every`` / ``each`` without a stated set),
   - optional per-type body-length bounds.

   Config in ``[tool.patdhlk-skills.lint]``; absent table = all checks
   off, each check individually toggleable. Lint reads a fresh
   needs.json (:need:`ADR_0006`), never the RST source. Violations
   exit 1 with JSON findings; exempt statuses (:need:`ISSUE_0018`) are
   honoured. Depends on the crate scaffold (:need:`ISSUE_0019`).

.. issue:: Verdict directives and pds verdict-check
   :id: ISSUE_0014
   :status: needs-triage
   :kind: feature

   *This was generated by AI during triage.*

   Today a triage or review judgment lives only in the ``:status:``
   flip and the git history. Make the judgment itself a gate-checkable
   artifact, per :need:`ADR_0015`: register a ``verdict`` need type
   (and role-map entry), have review skills author one verdict
   directive per judged need — derived ID ``VERDICT_<judged-id>``,
   fields ``:rubric:`` / ``:axes_failed:`` / ``:fingerprint:``,
   findings as body prose, linked to the judged need, living in
   ``spec/verdicts/``. Implement ``pds verdict-check``
   (:need:`ADR_0014`): derive pass (``axes_failed`` empty), recompute
   fingerprints from needs.json, and report four buckets — missing /
   failing / stale / malformed — with required types and rubrics read
   from ``[tool.patdhlk-skills.verdicts]`` (:need:`ADR_0016`). Wire
   into ``pds check --with-verdicts``. The payoff:
   ``ready-for-agent`` (:need:`ADR_0005`) can require a passing triage
   verdict on file, not just an edited status field. Depends on
   :need:`ISSUE_0019`.

.. issue:: pds next and pds status — deterministic backlog queries
   :id: ISSUE_0015
   :status: needs-triage
   :kind: improvement

   *This was generated by AI during triage.*

   An agent driving the backlog should never pick its own work by
   grepping around; it should ask the corpus. Implement two ``pds``
   verbs (:need:`ADR_0014`): ``pds next`` rebuilds needs.json
   (:need:`ADR_0006`) and emits the next actionable issue —
   ``status == ready-for-agent``, lowest ID first — as JSON (id,
   title, status, links); nothing ready is exit 0 with
   ``{"issue": null, "reason": "none-ready"}``. ``pds status`` emits
   per-status counts, giving /triage its "what needs attention"
   buckets (:need:`ADR_0005`) from one query path instead of ad-hoc
   jq per skill. Both verbs emit a versioned JSON shape; on a
   github-backend repo they exit 2 with the equivalent ``gh`` command
   (local-only v1, forward-compatible per :need:`ADR_0014`). Depends
   on :need:`ISSUE_0019`.

.. issue:: Stopping contracts and fail-closed judging in looping skills
   :id: ISSUE_0016
   :status: needs-triage
   :kind: improvement

   *This was generated by AI during triage.*

   Retrofit three authoring-discipline patterns into the skills that
   loop (/triage, /to-issues, /qa, /tdd, /review):

   - **Stopping and escalation section.** Each looping skill
     enumerates exactly which tool outputs mean success, stop, or
     escalate-to-human — so "stuck" is defined by observable state,
     not by the agent's patience.
   - **Stage-local validation.** A multi-step authoring loop validates
     only the step just produced; the whole-corpus gate
     (:need:`ADR_0007`) runs once at the end. Running the full gate
     per step stays red until everything exists and causes false
     escalation.
   - **Fail-closed judging.** Wherever a skill judges (triage
     recommendation, review verdict): uncertainty is a FAIL with a
     recorded finding, never a silent PASS; a failure with no finding
     is malformed.

   The ``pds`` exit contract (:need:`ADR_0014`) is the substrate for
   the first two patterns: exit 1 means "read the JSON, fix the
   corpus", exit 2 means "stop and escalate" — skills should branch on
   that, not on output prose.

.. issue:: pds dedup and pds search — retrieval over the corpus
   :id: ISSUE_0017
   :status: needs-triage
   :kind: improvement
   :implements: FEAT_0001

   *This was generated by AI during triage.*

   /qa already deduplicates against a fresh needs.json build
   (:need:`ISSUE_0005`), but it is the only issue-creating path that
   does, and it improvises the matching. Implement the shared engine
   as two ``pds`` verbs (:need:`ADR_0014`): ``pds dedup "<candidate>"``
   ranks existing needs by similarity and exits 1 when hits reach the
   threshold (a pre-filing gate); ``pds search "<query>"`` is the
   neutral query form (always exit 0). Both return ranked
   ``{id, title, score, engine}`` JSON. v1 engine is BM25 — in-memory,
   deterministic, offline; neural embeddings come later behind a cargo
   feature and ``--engine embed`` with lazy model download, without
   changing the JSON contract. Then point every issue-creating path
   (/qa, /to-issues, /triage) at ``pds dedup`` before filing. Depends
   on :need:`ISSUE_0019`.

.. issue:: Status-aware exemption for corpus checks
   :id: ISSUE_0018
   :status: needs-triage
   :kind: improvement

   *This was generated by AI during triage.*

   Any corpus-wide check added by :need:`ISSUE_0013` or
   :need:`ISSUE_0014` must be status-aware: needs whose status is
   terminal (``done``, ``wontfix``) are excluded from lint and verdict
   requirements before any check runs. Otherwise every new gate
   retroactively fails the whole closed backlog and the gates get
   disabled instead of trusted. Declare the exempt-status list once in
   ``ubproject.toml`` under ``[tool.patdhlk-skills.gate]``
   (:need:`ADR_0014`) so every ``pds`` check reads the same policy.

.. issue:: Scaffold the pds-cli crate and release pipeline
   :id: ISSUE_0019
   :status: done
   :kind: feature
   :links: ADR_0018, ADR_0019, ADR_0020

   *This was generated by AI during triage.*

   The foundation for :need:`ADR_0014`: a Rust workspace at ``cli/``
   (``pds-core`` lib + ``pds-cli`` bin per :need:`ADR_0018`),
   versioned in lockstep with the skills. Scope of this issue:

   - crate skeleton with the verb dispatcher, config loader for
     ``[tool.patdhlk-skills.*]`` (incl. role-map resolution and
     fail-on-drift, exit 2; discovery per :need:`ADR_0020`),
     needs.json reader, and the builder
     adapter (``ubc check`` + ``ubc build needs`` /
     ``uv run sphinx-build -W -b needs`` per :need:`ADR_0017`) —
     enough to ship ``pds build`` and ``pds check`` structural mode;
   - JSON-only stdout + 0/1/2 exit contract as shared verb plumbing
     (envelope per :need:`ADR_0019`);
   - CI: fmt, clippy, test on the Rust pipeline next to the sphinx
     gate;
   - release workflow: static binaries for ``aarch64-apple-darwin``,
     ``x86_64-apple-darwin``, ``x86_64-unknown-linux-musl``,
     ``aarch64-unknown-linux-musl`` to GitHub Releases, an
     ``install.sh``, and ``cargo publish`` of ``pds-cli`` (name
     verified available);
   - **macOS Gatekeeper**: downloaded binaries may be blocked when a
     quarantine attribute is set. Decide and implement the story:
     Developer ID codesigning + notarization in the release workflow
     when certificates are available; until then, at minimum ad-hoc
     signing (required on arm64), an ``install.sh`` that clears the
     quarantine attribute on the installed binary with the user's
     consent, and a documented ``cargo install`` fallback (locally
     built binaries are never quarantined).

   Implementation preferences (grilling of 2026-06-06): two
   increments — skeleton + CI first, release pipeline second.
   Dependency stack: clap (derive), serde + serde_json, toml;
   thiserror in ``pds-core``, anyhow only in the bin. TDD against
   fixture corpora (small needs.json + ubproject.toml pairs under
   ``cli/tests/fixtures/``, builder mocked via a fake-builder
   script), plus assert_cmd/trycmd end-to-end tests freezing the
   exit-code and stdout/stderr contract. Edition 2024, stable pinned
   via ``rust-toolchain.toml``, ``rust-version`` tracks the pin with
   no MSRV promise. Release increment uses cargo-dist per
   :need:`ADR_0018`.

   Windows is deferred (``cargo install`` is the escape hatch). The
   other ``pds`` verbs land via :need:`ISSUE_0013` /
   :need:`ISSUE_0014` / :need:`ISSUE_0015` / :need:`ISSUE_0017` on top
   of this scaffold.
