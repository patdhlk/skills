.. issue:: pds lint ŌĆö mechanized body lint
   :id: ISSUE_0013
   :status: done
   :kind: improvement

   *This was generated by AI during triage.*

   The strict gate (:need:`ADR_0007`, :need:`ADR_0017`) catches
   structural problems ŌĆö schema violations, broken links, duplicate
   IDs ŌĆö but says nothing about the substance of a need body.
   Implement body lint as a ``pds`` verb (:need:`ADR_0014`), runnable
   alone (``pds lint``) and as part of ``pds check``:

   - required sections per type (an ``arch-decision`` must carry
     Context / Decision / Consequences; an ``issue`` body must be
     non-trivial),
   - weasel-word check on ``req`` bodies (vague adjectives without a
     numeric or behavioral criterion),
   - unenumerated-quantifier check on ``req`` bodies (``all`` /
     ``every`` / ``each`` without a stated set),
   - optional per-type body-length bounds.

   Config in ``[tool.patdhlk-skills.lint]``; absent table = all checks
   off, each check individually toggleable. Lint reads a fresh
   needs.json (:need:`ADR_0006`), never the RST source. Violations
   exit 1 with JSON findings; exempt statuses (:need:`ISSUE_0018`) are
   honoured. Depends on the crate scaffold (:need:`ISSUE_0019`).

   **Agent brief** (triage of 2026-06-07; this section is the
   contract ŌĆö the prose above is context).

   *Category:* enhancement. *Summary:* add a ``lint`` verb to the
   ``pds`` CLI that mechanically checks need bodies from a fresh
   needs.json, standalone and inside ``pds check``.

   *Current behavior.* ``pds`` ships ``build`` / ``check`` /
   ``status`` / ``next``. The corpus reader exposes each need's
   ``content``; the queries module has a shared
   guardŌåÆbuildŌåÆload preamble; the config loader parses
   ``[tool.patdhlk-skills.*]`` tables with rawŌåÆvalidated structs and
   fail-on-drift validation; check-shaped verbs emit
   ``{"findings": [{check, severity, need, message}]}`` with
   ``need`` always null so far. ``[tool.patdhlk-skills.gate]`` has
   ``needs_json`` / ``sphinx_command`` only.

   *Desired behavior.*

   - ``pds lint``: rebuild needs.json (non-gating build), load the
     corpus, run every check enabled in
     ``[tool.patdhlk-skills.lint]``, emit one finding per violation
     (``check`` = ``lint:<rule>``, ``need`` = the offending need's
     ID, actionable ``message``), exit 1 when findings are
     non-empty, else 0. Absent lint table or no enabled checks =
     clean exit 0, empty findings. NO github-backend guard ŌĆö lint
     covers the whole spec corpus (reqs, ADRs), which exists on
     both backends.
   - The four rules, each independently toggleable by the presence
     of its config key: ``required-sections`` (per-directive list of
     bold lead-ins that must appear in the body, e.g. Context /
     Decision / Consequences); ``nontrivial-body`` (per-directive
     minimum body length); ``weasel-words`` (configurable word list
     + configurable target directives; flags listed words unless the
     sentence also carries a numeric or behavioral criterion ŌĆö v1
     may approximate "carries a criterion" as "contains a digit or
     a comparison word"; document the approximation in the rule's
     message); ``unenumerated-quantifiers`` (configurable quantifier
     list + target directives; flags quantifiers not followed by an
     enumeration cue such as ``of the`` / ``listed`` / ``in`` ŌĆö
     same documented-approximation latitude). Exact TOML key shapes
     are the agent's call within: keys reference DIRECTIVE names,
     validated against the declared ``[[needs.types]]`` (undeclared
     directive in lint config = config hard error, exit 2,
     ADR_0014); word lists live in config, not hardcoded (defaults
     are scaffolded by setup later ŌĆö ADR_0016 spirit).
   - **Status-aware exemption (implements :need:`ISSUE_0018`'s
     mechanism for lint):** ``exempt_statuses`` key on
     ``[tool.patdhlk-skills.gate]``, defaulting to
     ``["done", "wontfix"]`` when absent; needs whose status is in
     the list are skipped before any lint rule runs. Build it as
     shared plumbing the future verdict-check verb can reuse.
   - ``pds check`` runs lint automatically AFTER its builder gate
     succeeds, on the needs.json it just built (no second build),
     whenever the lint table is present ŌĆö no new flag. Builder
     findings and lint findings share the one findings array.
   - Dogfood: enable a minimal lint table in this repo's own
     ``ubproject.toml`` (required sections on ``arch-decision``,
     non-trivial ``issue`` bodies) and keep the gate green; update
     the setup skill's forward-looking-config reference so lint is
     documented as shipped, not future.

   *Acceptance criteria* (each independently verifiable):
   lint with no table exits 0 with empty findings; each of the four
   rules fires on a crafted fixture corpus and stays quiet on a
   clean one; exempt-status needs are skipped (fixture with a
   violating ``done`` need stays clean); lint config naming an
   undeclared directive exits 2 with a config error naming it;
   ``pds check`` on a lint-enabled project includes lint findings
   and exits 1 on violations; this repo's gate (``make strict``)
   stays green with the dogfood lint table enabled; fmt / clippy
   ``-D warnings`` / full test suite green; TDD discipline (tests
   first, behavior through the public interface).

   *Out of scope:* verdict directives and ``pds verdict-check``
   (:need:`ISSUE_0014`); semantic/AI judgment of body quality ŌĆö
   this verb is mechanical only; linting RST source files directly;
   any ``--with-lint`` style flags; ``search`` / ``dedup``;
   changing the JSON envelope or exit contract (ADR_0019).

.. issue:: Stopping contracts and fail-closed judging in looping skills
   :id: ISSUE_0016
   :status: needs-triage
   :kind: improvement

   *This was generated by AI during triage.*

   Retrofit three authoring-discipline patterns into the skills that
   loop (/triage, /to-issues, /qa, /tdd, /review):

   - **Stopping and escalation section.** Each looping skill
     enumerates exactly which tool outputs mean success, stop, or
     escalate-to-human ŌĆö so "stuck" is defined by observable state,
     not by the agent's patience.
   - **Stage-local validation.** A multi-step authoring loop validates
     only the step just produced; the whole-corpus gate
     (:need:`ADR_0007`) runs once at the end. Running the full gate
     per step stays red until everything exists and causes false
     escalation.
   - **Fail-closed judging.** Wherever a skill judges (triage
     recommendation, review verdict): uncertainty is a FAIL with a
     recorded finding, never a silent PASS; a failure with no finding
     is malformed.

   The ``pds`` exit contract (:need:`ADR_0014`) is the substrate for
   the first two patterns: exit 1 means "read the JSON, fix the
   corpus", exit 2 means "stop and escalate" ŌĆö skills should branch on
   that, not on output prose.

.. issue:: pds dedup and pds search ŌĆö retrieval over the corpus
   :id: ISSUE_0017
   :status: done
   :kind: improvement
   :implements: FEAT_0001
   :links: ADR_0021

   *This was generated by AI during triage.*

   /qa already deduplicates against a fresh needs.json build
   (:need:`ISSUE_0005`), but it is the only issue-creating path that
   does, and it improvises the matching. Implement the shared engine
   as two ``pds`` verbs (:need:`ADR_0014`): ``pds dedup "<candidate>"``
   ranks existing needs by similarity and exits 1 when hits reach the
   threshold (a pre-filing gate); ``pds search "<query>"`` is the
   neutral query form (always exit 0). Both return ranked
   ``{id, title, score, engine}`` JSON. v1 engine is BM25 ŌĆö in-memory,
   deterministic, offline; neural embeddings come later behind a cargo
   feature and ``--engine embed`` with lazy model download, without
   changing the JSON contract. Then point every issue-creating path
   (/qa, /to-issues, /triage) at ``pds dedup`` before filing. Depends
   on :need:`ISSUE_0019`.

   *This was generated by AI during triage.*

   **Agent Brief** (grilling session of 2026-06-07; the retrieval
   contract is pinned by :need:`ADR_0021` ŌĆö read it first).

   **Category:** enhancement. **Summary:** add ``pds search`` and
   ``pds dedup`` ŌĆö BM25 retrieval over a fresh corpus, with a
   normalized-score pre-filing gate on dedup.

   **Current behavior.** No ``pds`` verb ranks needs by similarity;
   /qa improvises matching with ad-hoc jq. The query-verb plumbing
   (backend guard ŌåÆ fresh build ŌåÆ corpus load) already exists as the
   shared preamble used by ``pds status`` / ``pds next``.

   **Desired behavior.**

   - ``pds search "<query>"``: shared preamble, then rank all need
     types by similarity; emit hits; always exit 0 (exit 2 reserved
     for tool/config errors, e.g. an empty query string).
   - ``pds dedup "<candidate>"``: same ranking; verdict
     ``duplicate`` and exit 1 iff at least one **issue-typed** hit
     scores Ōēź threshold; non-issue hits are listed but never gate.
   - JSON per :need:`ADR_0019` / :need:`ADR_0021`: top-level
     ``schema``, ``verb``, ``engine: "bm25"``; ``hits`` descending
     by ``score`` with deterministic id tie-break, capped at a small
     constant (e.g. 10); dedup adds ``threshold`` and ``verdict``.
     Hits are not findings-shaped ŌĆö dedup is a query verb that
     gates, not a check-shaped verb.
   - Engine: hand-rolled Okapi BM25 (``k1 = 1.2``, ``b = 0.75`` as
     code constants) in a pds-core retrieval module of pure
     functions over the loaded corpus, lib-testable without a
     builder ŌĆö mirror the existing pure-query module's shape. Zero
     new dependencies.
   - Tokenization: lowercase, split on non-alphanumerics, drop
     empties; no stemming, no stopword list; digit tokens kept (need
     ids in bodies become searchable). Field weighting by token
     repetition: title ├Ś3, tags ├Ś2, content ├Ś1 (code constants).
   - ``score`` is the 0ŌĆō1 self-score ratio per :need:`ADR_0021`; a
     document matched against its own text scores Ōēł 1.0.
   - Config: ``threshold`` (float, 0ŌĆō1) under
     ``[tool.patdhlk-skills.dedup]``; ``--threshold`` flag overrides.
     Pick the shipped default by eyeballing real hits against this
     repo's corpus and record the chosen value in the config docs.
   - Degenerate cases: empty corpus ŌåÆ empty hits, ``unique``,
     exit 0; query whose every token is unknown to the corpus ŌåÆ
     empty hits, ``unique``, exit 0.

   **Acceptance criteria.**

   - Dogfood: ``pds search "lint gate"`` on this repo exits 0 and
     ranks gate-related needs; ``pds dedup`` fed a near-copy of an
     existing open issue body exits 1 with ``verdict: duplicate``;
     fed novel text it exits 0 with ``verdict: unique``.
   - A non-issue need scoring above threshold does not flip the
     verdict (unit test).
   - A title match outranks an equivalent content match (unit test).
   - Self-match normalization Ōēł 1.0 within float tolerance (unit
     test).
   - Tie-break and ranking are deterministic across runs.
   - assert_cmd/trycmd end-to-end tests freeze both verbs' exit
     codes and stdout shapes, as for the existing verbs.
   - cargo fmt, clippy, and test pass; CLAUDE.md's backlog-verbs
     paragraph mentions the two new verbs.

   **Out of scope.**

   - Wiring /qa, /to-issues, /triage to ``pds dedup`` ŌĆö
     :need:`ISSUE_0021`.
   - The embed engine, ``--engine`` flag, and model download.
   - A GitHub-backend driver (guard exits 2 with a ``gh`` hint, as
     the existing query verbs do).
   - Stemming, stopwords, true BM25F, configurable BM25 parameters.
